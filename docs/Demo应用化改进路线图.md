# 瑕果智选 MVP Demo 应用化改进路线图

> 目标：不追求完整电商功能，而是把当前可演示 MVP 改造成一个有前端、有后端、有 UI、有数据流、有 AI 质量边界的“应用基本形态”。  
> 重点：用产品经理思维和 AI 研发技术基础，把项目从“功能展示”升级为“可讲清、可运行、可评估、可迭代”的 AI 原生产品 Demo。

## 1. 当前状态判断

当前 Demo 已经跑通一条最小闭环：

```text
农户提交苹果图片和基础信息
-> AI Agent 初步识别瑕疵并分级
-> 系统生成上架说明和消费者文案
-> 人工复核高风险商品
-> 低风险商品进入消费者页
-> 消费者模拟下单或反馈
-> 反馈和误判进入坏例池
```

已具备的基础能力：

| 模块 | 当前能力 | 应用化判断 |
| --- | --- | --- |
| 农户上架 | 上传/选择图片，填写基础信息，触发智能分级 | 已有核心任务入口，但还需要更清楚的页面状态和数据保存 |
| AI 分级 | Mock + Qwen-VL API 两种模式，输出结构化 JSON | 已有 AI 能力入口，但还需要提示词版本、输出校验、失败回退 |
| 分级报告 | 展示等级、瑕疵、置信度、复核状态、JSON | 已有可解释结果，但还需要面向不同角色的展示层级 |
| 消费者页 | 商品列表、详情、模拟下单、反馈 | 已有信任验证入口，但还需要更像真实商品页的 UI 和行为数据 |
| 运营后台 | 复核队列、商品管理、坏例池 | 已有后台雏形，但还需要操作日志、复核原因和状态流转 |
| Eval | 12 张样本批量评测 | 已有评测入口，但还不是正式人工标注集和指标体系 |

结论：下一步不是继续堆功能，而是做“应用化补强”。应用化的核心不是加支付、物流、账号体系，而是把页面、接口、数据对象、AI 调用、评测、人工复核和 UI 状态组织成一个稳定 Demo。

## 2. 产品定位收束

立项书的长期目标是“智能定级 + 多渠道销售 + 助农传播”的智能分级电商平台。但 Demo 阶段要收束为：

> 瑕果智选是一个面向苹果瑕疵果的 AI 辅助分级与可信上架 Demo。它帮助农户低门槛生成商品草稿，帮助平台拦截食品安全风险，帮助消费者理解为什么便宜、是否可买、出了问题如何处理。

本阶段明确不做：

| 暂不做 | 原因 |
| --- | --- |
| 真实支付 | 会引入合规、退款、财务和履约复杂度，不能直接证明分级信任是否成立 |
| 真实物流 | 当前验证重点是可售判断和信任说明，不是履约能力 |
| 多水果品类 | 不同水果分级标准差异大，过早扩展会稀释验证 |
| 完整账号权限 | Demo 阶段可用角色切换模拟，先不做注册、登录、认证 |
| 训练自有模型 | 当前样本不足，先验证通用视觉模型 + 规则 + 人工复核是否能跑通 |

学习点：这体现的是 MVP 收敛。产品经理不是把所有愿景都塞进第一版，而是在约束下选择最能验证核心假设的路径。

## 3. 应用化目标

本路线图的验收目标分为四层。

| 层级 | 目标 | 验收方式 |
| --- | --- | --- |
| 可运行 | 本地打开或启动服务后，完整跑通上传、分级、复核、上架、反馈、坏例 | 演示 5 分钟内不依赖口头补充 |
| 可理解 | 页面 UI 能让用户知道当前在哪一步、为什么这么判断、下一步能做什么 | 每个关键页面有清楚的主任务和状态 |
| 可追溯 | 每个商品能追溯图片、AI 输出、人工复核、消费者反馈、坏例记录 | 后台能查看商品生命周期 |
| 可评估 | 能用小样本 Eval 判断 AI 输出是否可用，尤其高风险漏判 | 输出指标表和失败样本列表 |

学习点：AI 产品 Demo 不是“页面能点”就够了。它还要能说明输入、链路、输出、失败处理和验证方法。

## 4. 总体路线

建议分 5 个阶段推进，每个阶段都可以独立验收。

```text
阶段 1：重构信息架构和 UI 壳
阶段 2：补齐前后端应用骨架
阶段 3：产品化 AI Agent 链路
阶段 4：建立 Eval、护栏和坏例闭环
阶段 5：整理作品化交付材料
```

优先级原则：

| 优先级 | 做什么 | 不做什么 |
| --- | --- | --- |
| P0 | 让核心 Demo 像应用一样跑起来 | 不做完整商城 |
| P1 | 让 AI 输出可解释、可复核、可回退 | 不追求模型训练 |
| P2 | 让项目能作为 PM 作品被讲清楚 | 不追求生产级工程 |

## 5. 阶段 1：重构信息架构和 UI 壳

### 目标

把当前页面从“功能区堆叠”整理成明确的三端应用结构：

```text
农户端：上架瑕疵果
消费者端：浏览可信瑕疵果
运营端：复核、规则、坏例、Eval
```

### 建议页面

| 页面 | 角色 | 主要任务 | 页面状态 |
| --- | --- | --- | --- |
| 应用首页 / 工作台 | 全部 | 选择进入农户端、消费者端、运营端 | 空状态、Demo 数据状态、API 状态 |
| 农户上架页 | 农户 | 上传图片、填写基础信息、触发 AI 分级 | 默认、上传中、分级中、分级失败 |
| AI 分级结果页 | 农户 | 查看等级、价格建议、是否复核、确认上架 | 可上架、需复核、禁售、低置信度 |
| 商品列表页 | 消费者 | 浏览已上架瑕疵果 | 有商品、无商品、筛选无结果 |
| 商品详情页 | 消费者 | 理解瑕疵、价格、安全说明、售后说明 | 可下单、已反馈、商品下架 |
| 运营工作台 | 运营 | 查看复核队列、商品状态、坏例和 Eval | 待处理、已处理、风险提示 |
| 复核详情页 | 运营 | 查看 AI 输出和人工处理 | 通过、驳回、要求补图 |
| 坏例池页 | 运营/PM | 记录失败样本和修复动作 | 待复盘、已修复、进入回归 |
| Eval 页 | PM/运营 | 批量评测样本，查看指标 | 未运行、运行中、完成、失败 |

### UI 改进重点

1. 顶部导航改为“三端 + Eval”的清楚结构。
2. 页面使用统一状态标签：`草稿`、`AI 已分级`、`待复核`、`已上架`、`已驳回`、`已进入坏例`。
3. 农户端减少字段，突出“上传 -> 分级 -> 确认”三步。
4. 消费者端把“为什么便宜”“是否影响食用”“平台如何兜底”放在商品详情页核心位置。
5. 运营端用表格和详情抽屉承载复核，不做花哨展示。

### 学习点：前端页面不是截图

产品经理描述前端时，不能只说“做一个页面”。要说清：

| 要素 | 在本项目中的例子 |
| --- | --- |
| 页面入口 | 从工作台进入农户端、消费者端、运营端 |
| 关键组件 | 上传区、分级结果卡、商品卡、复核表格、坏例表 |
| 默认状态 | 没有商品时显示引导上传，而不是空白 |
| 加载状态 | AI 分级时显示“正在识别图片并生成说明” |
| 错误状态 | API 失败时提示回退 mock，不让演示中断 |
| 数据依赖 | 商品列表来自商品数据，复核队列来自高风险商品 |

这一阶段的验收标准：别人第一次打开 Demo，不需要你解释太多，就能知道这是一个三端应用，而不是一张混合功能页。

## 6. 阶段 2：补齐前后端应用骨架

### 目标

把当前静态状态和本地服务整理成更清楚的前后端结构。Demo 不需要复杂框架，但需要像应用一样有接口、数据对象和状态流转。

### 建议技术形态

保持轻量，不上重型技术栈：

```text
前端：index.html + styles.css + app.js
后端：Node.js server.js
数据：本地 JSON 文件或 localStorage，优先保证 Demo 稳定
AI：/api/grade 调用 Qwen-VL，失败 fallback mock
```

如果要再像“应用”，可以增加：

```text
data/products.json
data/reviews.json
data/bad_cases.json
data/eval_results.json
```

### 最小接口清单

| 能力 | 接口建议 | 触发页面 | 输入 | 返回 |
| --- | --- | --- | --- | --- |
| 健康检查 | `GET /api/health` | 工作台 | 无 | API 状态、模型配置状态 |
| AI 分级 | `POST /api/grade` | 农户上架页 | 图片、商品基础字段 | 分级 JSON |
| 创建商品 | `POST /api/products` | 分级结果页 | 商品草稿、AI 报告 | 商品 ID、状态 |
| 查询商品 | `GET /api/products` | 商品列表/后台 | 状态、关键词 | 商品列表 |
| 更新商品状态 | `PATCH /api/products/:id/status` | 复核详情 | 目标状态、原因 | 更新后商品 |
| 创建反馈 | `POST /api/feedback` | 商品详情页 | 商品 ID、反馈类型、内容 | 反馈记录 |
| 创建坏例 | `POST /api/bad-cases` | 复核/反馈 | 失败类型、原因、修复动作 | 坏例记录 |
| 运行 Eval | `POST /api/evals/run` | Eval 页 | 样本集配置 | 指标和样本结果 |

### 学习点：页面、接口、数据对象是最小可构建表达

产品经理知识库里强调，PRD 之外还要补三类材料：

| 材料 | 本项目怎么用 |
| --- | --- |
| 页面 | 用户在哪里上传、查看、复核、反馈 |
| 接口 | 按钮背后调用什么能力，比如分级、创建商品、复核通过 |
| 数据对象 | 系统保存什么，比如商品、分级报告、复核记录、坏例 |

如果只做页面，没有接口和数据对象，Demo 很容易停留在“静态展示”。如果补齐这三类材料，就能向“应用基本形态”迈进。

这一阶段的验收标准：刷新页面后数据不乱，商品状态能流转，复核和坏例能追溯，接口失败有提示。

## 7. 阶段 3：产品化 AI Agent 链路

### 目标

把 AI 从“调用一次视觉模型”升级为“有上下文、有输出契约、有规则兜底、有失败回退的产品链路”。

### 推荐 Agent 工作流

```text
1. 接收图片和商品基础字段
2. 读取瑕疵分级规则
3. 调用视觉模型识别可见瑕疵
4. 将模型结果映射到业务等级
5. 执行硬性护栏：腐烂、霉变、破皮渗液、低置信度
6. 生成农户解释和消费者说明
7. 输出固定 JSON
8. 前端展示结果，必要时进入人工复核
```

### 上下文材料拆分

不要只把所有内容塞进一个 prompt。建议分成四类上下文：

| 上下文 | 内容 | 用途 |
| --- | --- | --- |
| 任务说明 | 你是瑕疵果分级助手，只做初判，不做最终食品安全承诺 | 限制 AI 角色 |
| 分级规则 | fresh=A，scab=B，bruise=C，rot=blocked | 约束业务判断 |
| 输出契约 | 必须返回固定 JSON 字段 | 保证前端可解析 |
| 护栏规则 | 腐烂、霉变、破皮渗液、低置信度必须复核 | 防止高风险绕过 |

### 输出契约

AI 输出必须稳定为：

```json
{
  "fruit_type": "apple",
  "defect_type": "bruise_defect",
  "defect_label": "轻微碰伤",
  "business_defect": "外观瑕疵，需谨慎复核",
  "grade": "C",
  "confidence": 0.76,
  "edible_safety": "uncertain",
  "safety_label": "需人工复核后判断",
  "price_suggestion": "市场价 50%-65%",
  "farmer_explanation": "图片中疑似存在碰伤区域，建议平台复核后上架。",
  "consumer_copy": "该商品需复核后展示给消费者。",
  "review_required": true,
  "risk_flags": ["bruise_review_required"],
  "next_action": "manual_review"
}
```

### 失败回退

| 失败情况 | 产品处理 |
| --- | --- |
| API Key 缺失 | 自动切到 mock，并提示“当前为演示模式” |
| 模型超时 | 保留农户表单，允许重试，不清空数据 |
| JSON 解析失败 | 后端尝试修复；仍失败则进入人工复核 |
| 置信度低 | 不直接上架，进入复核队列 |
| 高风险识别 | 强制复核，不展示给消费者 |

### 学习点：Context Engineering

Prompt Engineering 关注“一次输入怎么写”，Context Engineering 关注“长期可复用的上下文怎么组织”。在本项目中，不应该只写一句“帮我识别苹果瑕疵”，而是把 AI 需要的规则、字段、输出格式和风险边界结构化。

### 学习点：Agentic 不等于聊天助手

这个 Demo 的 Agent 不应该表现成聊天框，而应该是一个任务链路：

```text
看图 -> 对规则 -> 输出结构化报告 -> 触发复核或上架
```

AI 的价值不是“能聊天”，而是进入真实业务流程，同时被规则和人工控制。

这一阶段的验收标准：AI 输出 100% 可解析；高风险样本不会直接上架；API 失败时 Demo 不中断。

## 8. 阶段 4：建立 Eval、护栏和坏例闭环

### 目标

让 Demo 不只是“看起来能识别”，而是能说明 AI 质量如何评估、风险如何拦截、错误如何回流。

### Eval 样本集

先从 12 张扩展到 40 张苹果样本：

| 类别 | 数量 | 标注重点 |
| --- | --- | --- |
| fresh | 10 | 是否可上架，是否误判为瑕疵 |
| scab_defect | 10 | 是否判断为外观瑕疵，是否能生成可信说明 |
| bruise_defect | 10 | 是否触发复核，等级是否过高 |
| rot_defect | 10 | 是否 100% 拦截，不得进入消费者页 |

### 人工标注字段

```json
{
  "sample_id": "apple_001",
  "image_path": "assets/apple_samples/rot_defect/P_REAL_0001.png",
  "human_defect_type": "rot_defect",
  "human_grade": "blocked",
  "food_safety_risk": true,
  "review_required": true,
  "expected_action": "manual_review",
  "note": "疑似腐烂，不允许直接上架"
}
```

### 指标体系

| 指标 | 含义 | Demo 阶段目标 |
| --- | --- | --- |
| JSON 可解析率 | AI 输出能否被系统稳定读取 | 100% |
| 高风险复核召回率 | rot/霉变/破皮样本是否进入复核 | 100% |
| 腐烂风险漏判率 | 高风险样本被判断可上架的比例 | 0 |
| 等级一致率 | AI 等级与人工等级一致比例 | 先达到 70% |
| 文案合规率 | 消费者文案是否避免绝对安全承诺 | 100% |
| 坏例修复率 | 坏例是否有修复动作并进入回归 | 80% 以上 |

### 护栏规则

必须写成系统规则，而不是只依赖模型自觉：

```text
如果 defect_type 包含 rot/mold/leak/broken_skin，review_required 必须为 true。
如果 confidence < 0.7，review_required 必须为 true。
如果 edible_safety = risk，next_action 不能是 confirm_listing。
如果 consumer_copy 包含“绝对安全”“保证无风险”等表达，必须重写。
```

### 坏例回流

```text
AI 误判 / 人工驳回 / 用户负反馈
-> 进入坏例池
-> 标注失败类型和根因
-> 更新规则或 prompt
-> 加入 Eval 回归样本
-> 再次运行评测
```

### 学习点：Evals、Guardrails、人工复核不是一回事

| 概念 | 解决什么 | 本项目例子 |
| --- | --- | --- |
| Eval | 判断 AI 表现是否达标 | 40 张样本批量跑分 |
| Guardrails | 拦住不该发生的风险 | 腐烂样本不得上架 |
| 人工复核 | 处理责任重、机器不确定的场景 | 低置信度和高风险样本交给运营 |

这三个合在一起，才是 AI 产品质量治理闭环。

这一阶段的验收标准：Eval 页面能输出指标，失败样本能进入坏例池，坏例能被标记为“已修复/待修复/进入回归”。

## 9. 阶段 5：整理作品化交付材料

### 目标

让这个项目不只是能运行，还能作为产品经理作品讲清楚。

### 建议交付物

### 10 分钟讲解结构

```text
1 分钟：问题定义，瑕疵果不是没人要，而是缺少可信分级和销售说明
2 分钟：MVP 边界，只做苹果、四类瑕疵、模拟下单、不做完整电商
2 分钟：演示农户上传、AI 分级、确认上架
2 分钟：演示消费者如何理解瑕疵和下单
1 分钟：演示运营复核、坏例池和 Eval
1 分钟：解释 AI 与人工边界
1 分钟：讲下一步路线和指标
```

### 学习点：产品经理项目表达不只是功能清单

面试或汇报时少说“我做了哪些页面”，多说：

| 表达方向 | 本项目怎么讲 |
| --- | --- |
| 问题定义 | 核心阻碍是信任，不只是识别 |
| 取舍判断 | 先做苹果单品类，不做完整商城 |
| AI 边界 | AI 初判，人做最终责任判断 |
| 质量治理 | Eval + 护栏 + 人工复核 + 坏例回流 |
| 数据指标 | 上架完成率、复核触发率、漏判率、文案合规率 |

这一阶段的验收标准：不用打开代码，也能通过文档看懂项目目标、边界、架构、AI 链路和下一步计划。

## 10. 7 天执行排期

### Day 1：UI 信息架构整理

产出：

- 三端导航结构。
- 页面清单。
- 核心状态标签。
- 商品状态流转图。

验收：

- 用户能清楚区分农户端、消费者端、运营端。
- 首页不是宣传页，而是可操作工作台。

### Day 2：页面状态补齐

产出：

- 农户上架页的上传中、分级中、失败、成功状态。
- 商品列表空状态和筛选状态。
- 复核队列空状态和风险排序。

验收：

- 不同状态下页面不空白、不混乱。
- API 失败时不丢失表单内容。

### Day 3：接口和数据对象整理

产出：

- `Product`、`GradingReport`、`Review`、`Feedback`、`BadCase`、`EvalRun` 数据结构。
- 最小接口清单。
- 状态流转规则。

验收：

- 每个页面动作都有数据落点。
- 每个关键状态都能追溯来源。

### Day 4：AI Agent 链路产品化

产出：

- Prompt/Context 拆分。
- 输出 JSON Schema。
- API 失败回退逻辑。
- 高风险护栏规则。

验收：

- 真实 API 和 mock 都能返回统一字段。
- 腐烂/低置信度样本不会进入消费者页。

### Day 5：Eval 与坏例闭环

产出：

- 40 张样本标注模板。
- Eval 指标表。
- 坏例类型和修复动作枚举。

验收：

- 能运行一次批量评测。
- 失败样本能一键或手动进入坏例池。

### Day 6：应用化 UI 打磨

产出：

- 统一视觉层级。
- 商品卡片、复核表格、分级报告卡样式统一。
- 消费者详情页强化信任信息。

验收：

- 页面看起来像一个应用，而不是临时拼接页。
- 关键信息优先级清楚。

### Day 7：作品化文档和演示脚本

产出：

- 运行说明。
- 演示讲稿。
- 项目复盘说明。
- 下一步路线说明。

验收：

- 10 分钟内能讲清项目。
- 能回答“为什么不做完整电商”“AI 错了怎么办”“如何验证效果”。

## 11. 最小成功标准

这一轮不以功能数量衡量成功，而以 8 个问题是否被回答衡量：

1. 农户是否能在 3 分钟内完成一次商品草稿生成？
2. AI 是否能输出稳定 JSON？
3. 高风险样本是否一定进入复核？
4. 消费者是否能理解为什么便宜、是否可买、出了问题怎么办？
5. 运营是否能看到复核原因并记录处理结果？
6. 坏例是否能回到规则或 Eval？
7. 页面是否有加载、空、错、成功等基本应用状态？
8. 文档是否能讲清目标、边界、数据、接口、AI 链路和质量治理？

## 12. 一句话路线总结

下一阶段不要把瑕果智选扩成完整商城，而要把当前 Demo 打磨成一个“可信瑕疵果分级与上架应用”：前端有清晰三端 UI，后端有接口和数据对象，AI 有上下文、输出契约和护栏，运营有复核和坏例闭环，PM 有指标和讲解材料。
