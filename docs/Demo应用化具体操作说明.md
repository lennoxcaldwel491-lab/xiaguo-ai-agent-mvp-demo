# 瑕果智选 MVP Demo 应用化具体操作说明

## 0. 使用目标

这份说明用于把当前“可演示 Demo”继续打磨成一个更像真实应用的 AI 原生产品 Demo。

本阶段不做完整电商，不做真实支付、真实物流、多水果品类、账号权限和自训练模型。重点是把当前苹果瑕疵果 MVP 做到：

- 前端结构清楚
- 后端接口清楚
- 数据对象清楚
- AI 调用链路清楚
- 人工复核和坏例回流清楚
- Eval 和质量指标清楚
- 10 分钟内可以讲明白

## 1. 应用化验收目标

完成后需要能回答 8 个问题：

1. 农户是否能在 3 分钟内完成一次商品草稿生成？
2. AI 是否能输出稳定 JSON？
3. 高风险样本是否一定进入复核？
4. 消费者是否能理解为什么便宜、是否可买、出问题怎么办？
5. 运营是否能看到复核原因并记录处理结果？
6. 坏例是否能回到规则或 Eval？
7. 页面是否有加载、空、错、成功等基本应用状态？
8. 文档是否能讲清目标、边界、数据、接口、AI 链路和质量治理？

## 2. 阶段 1：重构信息架构和 UI 壳

### 目标

把当前混合功能页整理成三端应用结构：

```text
农户端：上架瑕疵果
消费者端：浏览可信瑕疵果
运营端：复核、规则、坏例、Eval
```

### 具体操作

1. 新增或调整首页为“应用工作台”，不要做宣传页。
2. 工作台提供 3 个主要入口：
   - 农户上架
   - 消费者浏览
   - 运营后台
3. 顶部或侧边导航改成清晰结构：
   - 工作台
   - 农户端
   - 消费者端
   - 运营端
   - Eval
   - AI 设置
4. 为所有商品统一状态标签：
   - `draft`：草稿
   - `ai_checked`：AI 已分级
   - `pending_review`：待复核
   - `listed`：已上架
   - `rejected`：已驳回
   - `bad_case`：已进入坏例
5. 农户端页面突出 3 步：
   - 上传/选择图片
   - 填写基础信息
   - 开始 AI 分级
6. 消费者详情页突出 3 个核心问题：
   - 为什么便宜
   - 是否影响食用
   - 不满意怎么办
7. 运营端用表格承载复核队列、商品列表、坏例池和 Eval 结果。

### 产出物

- 应用工作台页面
- 三端导航结构
- 商品状态标签体系
- 商品状态流转图

### 验收标准

- 第一次打开页面的人能看懂这是三端应用，而不是功能堆叠页。
- 用户能清楚知道当前在哪一端、下一步做什么。
- 每个关键页面都有明确主任务和状态提示。

## 3. 阶段 2：补齐前后端应用骨架

### 目标

让 Demo 不只是静态页面，而是有接口、有数据对象、有状态流转的轻量应用。

### 当前推荐技术形态

```text
前端：index.html + styles.css + app.js
后端：Node.js server.js
AI：POST /api/grade 调用 Qwen-VL，失败 fallback mock
数据：localStorage + 可选 data/*.json
```

### 具体操作

1. 保留当前轻量结构，不急着迁移 Next.js。
2. 整理最小接口清单：
   - `GET /api/health`
   - `POST /api/grade`
   - `POST /api/products`
   - `GET /api/products`
   - `PATCH /api/products/:id/status`
   - `POST /api/feedback`
   - `POST /api/bad-cases`
   - `POST /api/evals/run`
3. 先不一定全部实现为真实接口，但文档里必须写清楚每个页面动作对应哪个接口。
4. 整理最小数据对象：
   - `Product`
   - `GradingReport`
   - `Review`
   - `Feedback`
   - `BadCase`
   - `EvalRun`
5. 给商品状态流转增加记录：
   - 谁触发
   - 什么时间
   - 从什么状态到什么状态
   - 原因是什么

### 产出物

- `docs/应用化技术需求清单.md`
- 接口清单
- 数据对象清单
- 商品状态流转规则

### 验收标准

- 每个按钮背后都有明确的数据动作。
- 每个关键状态都能追溯来源。
- API 失败不会清空用户已填写内容。
- 刷新页面后关键演示数据不丢失。

## 4. 阶段 3：产品化 AI Agent 链路

### 目标

把 AI 从“一次视觉模型调用”升级为有上下文、有输出契约、有护栏、有失败回退的产品链路。

### Agent 工作流

```text
1. 接收图片和农户基础字段
2. 读取瑕疵分级规则
3. 调用 Qwen-VL 识别可见瑕疵
4. 将模型结果映射到业务等级
5. 执行硬性护栏
6. 生成农户解释和消费者说明
7. 输出固定 JSON
8. 前端根据 next_action 决定上架或复核
```

### 具体操作

1. 拆分 AI 上下文为 4 类：
   - 任务说明
   - 分级规则
   - 输出 JSON Schema
   - 护栏规则
2. 固定输出字段：
   - `fruit_type`
   - `defect_type`
   - `defect_label`
   - `business_defect`
   - `grade`
   - `confidence`
   - `edible_safety`
   - `safety_label`
   - `price_suggestion`
   - `farmer_explanation`
   - `consumer_copy`
   - `review_required`
   - `risk_flags`
   - `next_action`
3. 后端对模型输出做归一化：
   - 非法 `defect_type` 归入高风险或人工复核
   - 非法 `grade` 修正为规则允许值
   - `confidence < 0.7` 强制复核
   - `edible_safety = risk` 强制复核
4. API 失败时处理：
   - API Key 缺失：fallback mock
   - 模型超时：保留表单，允许重试
   - JSON 解析失败：进入人工复核
   - 高风险识别：不展示给消费者

### 产出物

- `docs/AI_Agent设计.md`
- Prompt/Context 拆分
- 输出 JSON Schema
- API 失败回退逻辑
- 护栏规则表

### 验收标准

- Mock 和真实 API 都返回统一字段。
- AI 输出 100% 可被前端解析。
- 腐烂、霉变、破皮渗液、低置信度样本不会直接上架。
- API 失败时 Demo 不会中断。

## 5. 阶段 4：建立 Eval、护栏和坏例闭环

### 目标

让 Demo 能说明 AI 质量如何评估、风险如何拦截、错误如何回流。

### 具体操作

1. 从 12 张样本扩展到 40 张苹果样本：
   - `fresh`：10 张
   - `scab_defect`：10 张
   - `bruise_defect`：10 张
   - `rot_defect`：10 张
2. 建立人工标注文件：

```text
eval/apple_eval_set.json
```

3. 每个样本标注：
   - `sample_id`
   - `image_path`
   - `human_defect_type`
   - `human_grade`
   - `food_safety_risk`
   - `review_required`
   - `expected_action`
   - `note`
4. Eval 指标包括：
   - JSON 可解析率
   - 高风险复核召回率
   - 腐烂风险漏判率
   - 等级一致率
   - 文案合规率
   - 坏例修复率
5. 建立坏例回流流程：

```text
AI 误判 / 人工驳回 / 用户负反馈
-> 进入坏例池
-> 标注失败类型和根因
-> 更新规则或 Prompt
-> 加入 Eval 回归样本
-> 再次运行评测
```

### 产出物

- `eval/apple_eval_set.json`
- `docs/Eval与质量治理.md`
- Eval 指标表
- 坏例类型枚举
- 坏例修复动作枚举

### 验收标准

- 能运行一次批量评测。
- 能输出指标表。
- 失败样本能进入坏例池。
- 坏例能标记为待修复、已修复、进入回归。
- 腐烂风险漏判率必须为 0。

1 分钟：演示运营复核、坏例池和 Eval
1 分钟：解释 AI 与人工边界

## 9. 最小成功标准

这一轮完成后，项目应该达到：

```text
一个人打开 Demo
-> 能看懂三端结构
-> 能跑通农户上架
-> 能看到 AI 初判和人工复核
-> 能理解消费者信任说明
-> 能看到坏例和 Eval
-> 能通过文档理解数据、接口和 AI 质量治理
```

一句话：下一阶段不是把瑕果智选扩成完整商城，而是把当前 Demo 打磨成一个“可信瑕疵果分级与上架应用”。
