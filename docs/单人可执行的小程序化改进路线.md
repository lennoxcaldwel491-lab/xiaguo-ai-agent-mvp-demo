# 瑕果智选：单人可执行的小程序化改进路线

> 目标：基于当前 Demo、差距分析文档和产品经理知识库，筛选出一个人可以完成的行动，把瑕果智选从 Web Demo 推进为更完整、可实际操作、具备小程序上架准备度的产品。  
> 当前日期：2026-06-01  
> 推荐目标：先做“可体验、可审核、非真实交易”的小程序版本，再逐步补真实交易、资质、物流和试点。

## 1. 先定边界：一个人能做到什么

当前项目的长期愿景是“智能分级电商平台”，但一个人短期更适合推进的是：

```text
苹果单品类
-> AI 辅助分级
-> 农户生成商品草稿
-> 运营复核
-> 消费者查看可信商品页
-> 模拟下单/购买意向
-> 反馈和坏例回流
-> 小程序体验版/审核版
```

不建议一个人当前直接做：

| 不建议做 | 原因 |
| --- | --- |
| 真实支付 | 涉及资质、交易合规、退款、账务和售后责任 |
| 真实物流 | 需要外部物流资源、地址管理、发货和售后履约 |
| 多品类识别 | 分级规则、样本集、模型表现都会复杂化 |
| 自研 YOLO/UNet 模型 | 没有足够标注集前，训练结果难以验证 |
| B 端采购完整系统 | 需要真实加工企业需求、报价、起订量和供应链数据 |
| 正式规模化运营 | 需要农户、企业、平台、政府或媒体资源 |

一个人短期应该做：

| 可做方向 | 具体产出 |
| --- | --- |
| 产品规则 | 苹果分级规则、价格建议规则、复核规则 |
| 数据闭环 | 商品、分级报告、复核、反馈、坏例、Eval 数据结构 |
| AI 链路 | 结构化输出、失败回退、低置信度复核、高风险护栏 |
| 应用体验 | 三端 UI、页面状态、错误提示、操作日志 |
| 小程序化 | 用微信开发者工具或 Uni-app 迁移核心页面 |
| 上架准备 | 账号、类目、备案、隐私协议、服务说明、审核材料 |

## 2. 选择原则

从差距分析里筛任务时，用 5 条标准判断。

| 标准 | 判断问题 | 本项目应用 |
| --- | --- | --- |
| 单人可控 | 是否不依赖外部商家、政府、物流、支付资质 | 优先做规则、页面、数据、AI、文档 |
| 能提升完整度 | 是否让 Demo 更像真实应用 | 优先补状态、数据、接口、后台 |
| 能服务小程序上架 | 是否能直接迁移为小程序页面或审核材料 | 优先做小程序核心流程和说明文案 |
| 能体现 PM 能力 | 是否能讲清目标、边界、指标、风险 | 优先做 Eval、护栏、复核、坏例 |
| 不扩大风险 | 是否避免食品安全、支付、真实交易责任 | 先做模拟下单和购买意向，不做支付 |

产品经理知识库里的核心方法可以落成一句话：

> 不要只说“做一个小程序”，而要说清页面、接口、数据对象、AI 调用、失败状态、上线边界和验收标准。

## 3. 推荐产品形态

### 3.1 小程序版本定位

建议把小程序第一版定位为：

> 瑕果智选小程序体验版：用于展示苹果瑕疵果的 AI 辅助分级、可信商品说明、人工复核和购买意向收集，不承诺真实交易履约。

第一版避免写成：

> 瑕果智选正式电商平台，支持真实购买、支付和发货。

原因：真实交易会触发更高的资质、支付、食品安全、售后和平台审核要求。一个人可以先做“演示/试用/调研型小程序”，把核心价值跑通。

### 3.2 用户角色

保留 3 个角色，但都做轻量版：

| 角色 | 小程序第一版任务 | 是否必须登录 |
| --- | --- | --- |
| 农户 | 上传图片，填写基础信息，生成商品草稿 | 可先不做登录，用演示身份 |
| 消费者 | 查看商品说明，提交购买意向或顾虑反馈 | 可先不做登录 |
| 运营/项目方 | 复核高风险商品，查看坏例和 Eval | 可用隐藏入口或演示密码 |

### 3.3 核心页面

一个人可完成的小程序页面建议控制在 8 个以内。

| 页面 | 作用 | 当前 Web Demo 迁移来源 |
| --- | --- | --- |
| 首页/工作台 | 进入农户端、消费者端、运营端 | 当前导航和总览 |
| 农户上传页 | 图片上传、基础信息填写 | 农户上架端 |
| AI 分级结果页 | 展示等级、置信度、复核、价格建议 | 分级报告页 |
| 商品列表页 | 消费者浏览可展示商品 | 消费者商品列表 |
| 商品详情页 | 解释为什么便宜、是否可买、售后边界 | 商品详情 |
| 购买意向/反馈页 | 收集“愿意购买/担心安全/价格不够”等反馈 | 消费者反馈 |
| 运营复核页 | 查看待复核商品，通过/驳回/记录原因 | 复核队列 |
| 坏例与 Eval 页 | 展示失败样本、规则修复和评测结果 | 坏例池 + Eval |

## 5. 小程序上架路径

### 5.1 两条可选路径

| 路径 | 适合情况 | 优点 | 风险 |
| --- | --- | --- | --- |
| 原生微信小程序 | 你愿意直接写 WXML/WXSS/JS | 最贴近微信审核和能力 | 需要重写 Web 页面 |
| Uni-app | 你想保留类 Vue 的开发方式，后续可多端 | 更适合从 Web 思维迁移 | 需要搭建 Uni-app 工程 |

建议：如果目标是尽快上架体验版，优先原生微信小程序；如果目标是后续长期维护和多端，选 Uni-app。

### 5.2 上架前必须准备

以下事项不全是代码任务，但会影响小程序能否发布。

| 准备项 | 说明 | 一个人可做程度 |
| --- | --- | --- |
| 小程序账号 | 微信公众平台注册小程序账号 | 可做 |
| AppID | 微信开发者工具需要绑定 AppID | 可做 |
| 基础信息 | 名称、头像、简介、服务范围 | 可做 |
| 服务类目 | 如果涉及水果售卖，通常要选择生鲜/初级食用农产品等相关类目，以平台实际提示为准 | 可准备，可能需要主体资质 |
| 备案 | 小程序正式发布通常需要完成备案，具体以微信公众平台和监管要求为准 | 可准备，但需要主体信息 |
| 隐私协议 | 收集图片、联系方式、反馈时需要说明用途 | 可做 |
| 用户协议 | 说明演示版、非真实交易、AI 初判边界 | 可做 |
| 审核截图 | 上传、分级、商品详情、反馈、复核等页面截图 | 可做 |
| 测试账号/路径 | 告诉审核人员如何进入演示流程 | 可做 |
| 后端域名 | 正式小程序请求线上 API 需要合法域名和 HTTPS | 需要部署 |

重要边界：

> 如果你没有企业/个体工商户主体和相关经营资质，建议第一版不要做真实“卖水果下单支付”，而做“AI 分级展示 + 购买意向收集 + 项目调研/演示”。正式交易版应等主体、类目、备案、支付、售后和食品相关责任明确后再做。

### 5.3 小程序第一版不接支付的设计

把“购买”改成“购买意向”，更适合一个人推进。

| 原电商动作 | 小程序第一版替代 |
| --- | --- |
| 立即购买 | 我有购买意向 |
| 支付下单 | 留下需求和联系方式 |
| 订单详情 | 意向记录 |
| 物流状态 | 暂不支持真实履约 |
| 退款售后 | 反馈/顾虑收集 |

这样仍然能验证核心问题：

```text
用户是否看得懂瑕疵说明？
用户是否相信 AI + 平台复核？
用户是否愿意留下购买意向？
用户最担心安全、价格还是描述不清？
```

## 7. 一个人可以做的小程序页面 PRD

### 7.1 首页

目标：让用户立即知道这是“瑕疵果 AI 分级与可信上架”工具。

页面模块：

- 项目名称：瑕果智选
- 三个入口：我要上架、我要看看、运营复核
- 当前 Demo 数据：已上架、待复核、已反馈
- API 状态：演示模式/真实 AI 模式

验收：

- 用户 10 秒内知道能做什么。
- 不做大段宣传文案。

### 7.2 农户上传页

目标：让农户低门槛生成商品草稿。

字段：

- 图片
- 水果品类，默认苹果
- 产地
- 重量
- 采摘时间
- 期望售价
- 备注

状态：

- 未上传
- 上传中
- AI 分级中
- 分级失败
- 分级完成

验收：

- 表单字段不超过 6 个核心项。
- API 失败时不清空已填内容。

### 7.3 AI 分级结果页

目标：让农户理解 AI 初判，而不是盲目相信 AI。

展示：

- 建议等级
- 瑕疵类型
- 置信度
- 食用安全边界
- 价格建议
- 农户解释
- 是否需要复核

动作：

- 确认上架
- 提交复核
- 返回修改

验收：

- 高风险结果不能出现“确认上架”主按钮。
- 低置信度必须进入复核。

### 7.4 商品详情页

目标：降低消费者对瑕疵果的信任顾虑。

展示顺序：

1. 图片和价格
2. 为什么便宜
3. 是否影响食用
4. 瑕疵说明
5. AI 初判和人工确认状态
6. 产地与采摘信息
7. 平台兜底和反馈入口

动作：

- 我有购买意向
- 我还有顾虑
- 描述不清楚
- 价格不够有吸引力

验收：

- 不出现“绝对安全”“保证无风险”等表达。
- 明确写明 AI 是初判，食品安全最终需要人工复核。

### 7.5 运营复核页

目标：让高风险商品有人工兜底。

展示：

- 商品图片
- 农户信息
- AI 输出
- 风险标签
- 触发复核原因

动作：

- 通过上架
- 驳回
- 要求补图
- 标记坏例

验收：

- 复核必须填写原因。
- 驳回自动进入坏例池。

## 8. 产品经理知识点对应

| 知识点 | 在本项目中的用法 | 你要学会什么 |
| --- | --- | --- |
| MVP 收敛 | 先做苹果单品类，不做完整商城 | 会取舍 |
| 页面/接口/数据对象 | 把小程序拆成页面、API、业务对象 | 会把想法说成可构建对象 |
| 前端状态 | 加载、空、错、成功、低置信度、高风险 | 不把页面当截图 |
| 后端和数据 | 商品、报告、复核、反馈、坏例要保存 | 知道按钮背后要有数据 |
| AI 模型调用 | Qwen-VL + mock fallback | 知道 AI 如何进入产品 |
| Context Engineering | 规则、输出 Schema、护栏、示例分开组织 | 不只写一句 prompt |
| HAX 人机协同 | AI 初判，人复核高风险 | 知道控制权在哪里 |
| AI 质量治理 | Eval、护栏、坏例池 | 知道如何评估和改进 AI |
| 小程序上线 | 类目、备案、审核、隐私、服务说明 | 知道 Demo 到发布的差距 |

## 9. 最小可上架版本定义

如果目标是一个人尽快做出可提交审核的小程序，建议最小版本定义为：

```text
瑕果智选小程序 v0.1

定位：
苹果瑕疵果 AI 分级展示与购买意向收集工具。

包含：
- 农户上传苹果图片
- AI 生成分级建议
- 高风险进入复核
- 消费者查看商品说明
- 消费者提交购买意向/顾虑反馈
- 运营查看复核和坏例

不包含：
- 真实支付
- 真实物流
- 真实售后
```
