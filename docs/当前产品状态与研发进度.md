# 瑕果智选 AI Agent MVP 苹果版：当前产品状态与研发进度

更新时间：2026-05-28

## 1. 当前版本定位

当前版本是“瑕果智选 AI Agent MVP 苹果版”的可运行 Demo，目标是先验证单品类苹果的核心业务闭环：

农户上传或选择苹果图片，系统调用 AI 进行瑕疵识别和分级，生成结构化分级报告；低风险商品可进入消费者页，高风险或低置信度商品进入运营复核；运营可处理复核、维护坏例池，并通过 Eval 检查 AI 输出质量。

当前版本不再以作品展示页为主，而是按真实产品逻辑拆分为三个独立端口：

- 农户端：提交苹果图片并获取分级建议。
- 消费者端：查看已上架的瑕疵苹果商品和透明说明。
- 运营端：处理复核、维护规则、查看坏例和评测结果。

## 2. 当前产品逻辑

### 2.1 主流程

1. 用户进入首页，选择身份：农户、消费者或运营人员。
2. 农户端选择苹果样本或上传本地图片。
3. 农户填写产地、重量、采摘时间、期望售价、备注等信息。
4. 点击“开始智能分级”。
5. 系统进入 AI Agent 分级流程：
   - 识别苹果瑕疵类型。
   - 判断业务等级。
   - 给出食用安全提示。
   - 给出价格建议。
   - 判断是否需要人工复核。
   - 生成农户解释和消费者透明说明。
6. 如果是低风险商品，农户可确认上架，商品进入消费者端。
7. 如果是碰伤、疑似腐烂、低置信度或高风险样本，商品进入运营复核队列。
8. 运营人员可通过或驳回复核。
9. 驳回或误判样本可进入坏例池，用于后续优化规则和模型提示词。
10. 运营端可运行 Eval，检查 AI 输出是否满足当前验收标准。

### 2.2 当前苹果分级规则

| 数据集标签 | 业务含义 | 商品等级 | 处理方式 |
| --- | --- | --- | --- |
| fresh | 无明显瑕疵 | A | 可确认上架 |
| scab_defect | 果锈/疮痂斑 | B | 可确认上架 |
| bruise_defect | 轻微碰伤 | C | 进入人工复核 |
| rot_defect | 疑似腐烂 | blocked | 禁止直接上架，强制人工复核 |

## 3. 已完成的功能

### 3.1 农户端

- 支持选择内置苹果样本。
- 支持上传本地苹果图片。
- 支持查看运营要求补图/补充信息的任务，并重新提交复核。
- 支持填写产地、重量、采摘日期、期望售价和农户备注。
- 支持 Mock 模式和 API 模式。
- 支持调用真实视觉大模型接口进行图片分级。
- 展示 AI 分级结果，包括：
  - 瑕疵类型
  - 业务瑕疵描述
  - 商品等级
  - 置信度
  - 食用安全提示
  - 价格建议
  - 是否需要复核
  - 下一步动作
  - 风险标签
- 支持复制和下载结构化 JSON 报告。
- 支持确认上架或提交复核。

### 3.2 消费者端

- 只展示已经上架的商品。
- 展示商品图片、等级、产地、重量、价格。
- 展示透明说明：
  - 为什么便宜
  - 是否影响食用
  - 平台兜底说明
- 支持模拟下单。
- 支持消费者反馈：
  - 愿意购买
  - 担心食用安全
  - 说明不清楚
  - 价格不够有吸引力
  - 不接受这种瑕疵

### 3.3 运营端

- 支持查看待复核商品。
- 支持通过复核，使商品进入消费者端。
- 支持要求农户补图或补充基础信息。
- 支持驳回复核，并将问题样本加入坏例池。
- 支持查看商品状态表。
- 支持查看规则列表。
- 支持查看坏例池。
- 支持查看 AI 设置和后端契约配置。
- 支持运行批量 Eval。

### 3.4 数据持久化

- 浏览器本地会保存当前状态。
- 后端会将状态保存到 `data/app_state.json`。
- 当前保存的数据包括：
  - 当前 AI 报告
  - 商品列表
  - 复核列表
  - 用户反馈
  - Eval 结果
  - 坏例池
  - 更新时间

当前 Node.js 服务运行在：


已具备的主要接口：

| 接口 | 作用 |

| `POST /api/feedback` | 保存消费者反馈 |
| `POST /api/bad-cases` | 保存坏例 |
| `POST /api/evals/run` | 保存 Eval 运行结果 |

## 4. 当前 AI Agent 体现在哪里

当前版本的 Agent 不是一个聊天机器人，而是一个“任务型业务 Agent”。它体现在以下几个层面：

### 4.1 有明确任务

Agent 的任务是：

根据苹果图片和农户填写的信息，完成瑕疵识别、业务分级、食用风险判断、价格建议、复核判断和文案生成。

### 4.2 有输入上下文

Agent 使用的输入包括：

- 苹果图片
- 水果类型
- 产地
- 重量
- 采摘日期
- 期望售价
- 农户备注
- 当前业务规则

### 4.3 有结构化输出

Agent 不只是返回一段自然语言，而是返回固定 JSON 字段，例如：

- `fruit_type`
- `defect_type`
- `defect_label`
- `business_defect`
- `grade`
- `confidence`
- `edible_safety`
- `safety_label`
- `price_suggestion`
- `farmer_explanation`
- `consumer_copy`
- `review_required`
- `risk_flags`
- `next_action`

这些字段可以被前端页面、运营流程和后续数据分析继续使用。

### 4.4 有业务规则和护栏

当前 Agent 有硬性护栏：

- 疑似腐烂必须进入 `blocked`。
- 食用风险样本不能直接上架。
- 置信度低于 0.7 必须进入人工复核。
- 消费者文案不能出现绝对安全承诺。
- 高风险商品不能生成可购买式消费者文案。

这说明当前 AI 不是随便生成文字，而是在业务规则约束下完成任务。

### 4.5 有人工兜底

Agent 不直接决定所有商品能否售卖。它只做初步判断：

- 低风险商品可以进入上架确认。
- 碰伤、腐烂、低置信度、高风险样本必须进入运营复核。

这符合食品相关场景下“AI 辅助，人工最终把关”的产品边界。

### 4.6 有 Eval 和坏例回流

当前版本已经加入 Eval 和坏例池：

- Eval 用于检查 AI 输出质量。
- 坏例池用于记录错误、边界样本和规则不足。
- 后续可以根据坏例持续优化提示词、规则和模型选择。

这说明当前 Agent 已经开始具备“可评测、可复盘、可迭代”的形态。

## 6. 当前版本的局限

当前版本已经跑通 MVP，但还不是完整生产级产品，主要局限包括：

- 目前只做了苹果单品类。
- 图片识别能力依赖通用视觉大模型，还没有训练专用模型。
- 当前分级规则较粗，缺少瑕疵面积、软烂程度、批次数量等精细字段。
- 消费者端还没有真实订单、支付、库存和售后流程。
- 运营端的坏例池还只是基础记录，缺少状态流转。
- Eval 仍偏规则验证，还没有形成更完整的人工标注集。
- API Key 目前放在本地 `.env.local`，适合本地 MVP，不适合公开部署。

## 7. 下一阶段建议
