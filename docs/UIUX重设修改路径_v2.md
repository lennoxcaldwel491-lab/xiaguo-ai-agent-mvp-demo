# 瑕果智选 UI/UX 重设修改路径 v2

更新日期：2026-06-19  
用途：作为后续 Agent 修改前端的执行路线，不是视觉灵感清单。

## 1. 当前问题的根因

### 1.1 用户体验混乱

1. **用户任务与演示工具混在一起**  
   农户和消费者界面出现 `Mock 模式`、生成演示商品、重置、临时标注等内部概念。用户无法判断哪些是自己要完成的任务，哪些是项目演示开关。

2. **页面按“功能模块”组织，没有按“用户下一步”组织**  
   农户首页同时展示拍照、相册、Mock 标签、12 张样本和历史商品；运营首页先展示大量指标，真正需要处理的待办反而在下方。

3. **缺少稳定的流程定位**  
   用户知道当前是“农户端”，却不知道自己正处在拍照、填写信息、等待分级还是确认上架的哪一步。

4. **页面语言混杂**  
   `Consumer`、`Operations`、`Priority`、`Eval`、`AI Config` 与中文业务文案并存，产品像开发控制台，而不是面向果农和消费者的完整服务。

5. **角色切换是全局主操作，抢占每个页面的空间**  
   正常用户进入某个角色后不会频繁切换身份。当前顶栏把切换角色、演示数据、重置和业务主操作放在同一层级，导致主次失衡。

### 1.2 UI 单调、没有设计感

1. **所有内容都是同一种米白卡片**  
   首页、商品、指标、待办、表单和说明使用近似的白底、圆角、细边框和阴影。内容类型虽不同，视觉语气却完全相同。

2. **角色没有真正形成差异**  
   农户、消费者和运营只是内容不同，布局节奏、组件形态、图像比例和信息密度差异不够。

3. **首页风格落入常见 AI 模板**  
   暖米色背景、宋体大标题、红色印章和低饱和绿色是当前很常见的生成式页面组合，并没有真正来自“苹果分级”的操作和物料。

4. **苹果图像是资料，却没有成为视觉主角**  
   消费者商品图较小，农户样本图像排列像数据集浏览器。图像没有用来表达果面状态、缺陷位置和分级差异。

5. **等级与风险的视觉权重太弱**  
   A/B/C/禁售是产品最核心的信息，现在只表现为小标签，与产地、状态等普通信息几乎同级。

## 2. 新的产品结构

不再用一套“万能卡片页”承载全部功能，而是分成两种表面。

### 2.1 用户任务面（农户／消费者）

- 移动优先，宽度随屏幕自适应，不做“桌面中间放一台模拟手机”的展示框。
- 每个页面只有一个主任务。
- 不显示 Mock、Eval、API、生成演示数据和重置等开发操作。
- 演示开关移入独立的“演示工具”抽屉，默认折叠，只在 demo 环境出现。

### 2.2 内部工作面（运营／AI 质量）

- 桌面优先，使用稳定的侧边导航和工作区。
- 导航收敛为 5 组：总览、商品复核、用户反馈、AI 质量、系统与日志。
- 坏例、规则和 Eval 统一放入“AI 质量”，不再占用三个同级顶部标签。
- 首屏先出现需要人做决策的待办，指标放在其后或缩成一行摘要。

## 3. 三条主流程

### 3.1 农户端：4 步完成一件事

```text
拍照/选图 → 确认基本信息 → 查看分级结果 → 确认上架或提交复核
```

修改要求：

- 首屏只保留大型拍照入口、相册入口和一句拍摄指导。
- 12 张样本不对真实农户展示；放入演示工具抽屉。
- 页面顶部显示真实步骤：`1 选图 / 2 信息 / 3 分级 / 4 确认`。
- 分级结果使用“检验决策卡”，优先显示等级、风险、是否可食和下一步；JSON 与技术字段放入“查看技术详情”。
- 已上传商品改为独立的“我的苹果”页，不与拍照任务同屏。

### 3.2 消费者端：先信任，后意向

```text
商品浏览 → 理解瑕疵与价格 → 查看安全边界 → 提交购买意向/顾虑
```

修改要求：

- 商品图放大，让瑕疵部位成为主视觉，不再只是左侧缩略图。
- 列表卡只保留：商品名、等级、瑕疵类型、价格、重量和一句“为什么值这个价”。
- 详情页首屏回答四个问题：为什么便宜、能不能吃、适合怎么吃、平台如何处理风险。
- 等级不使用小胶囊表达，改为明确的等级色带。
- “提交购买意向”为唯一主操作；“我有顾虑”使用次级文本按钮。
- 去掉顶栏中的生成演示商品和重置。

### 3.3 运营端：先决策，后观察

```text
查看高优先级待办 → 进入单条复核 → 做出决策并记录原因 → 追踪质量趋势
```

修改要求：

- 左侧使用稳定导航，右侧显示页面标题、筛选和主工作区。
- 首屏放“今天必须处理”，按食安风险、等待时间和用户影响排序。
- KPI 从 7 张相同卡片收敛为 3–4 个有决策意义的指标：待处理、高风险、超时、今日已完成。
- 复核详情采用三栏结构：图像与标记 / AI 结论与规则 / 人工决策。
- “通过”、“补资料”、“驳回”形成清晰的成功、警告和危险层级，不再都是近似白色按钮。
- 只在质量页使用 Eval、Trace 等专业词，并提供中文说明。

## 4. 新的视觉方向

### 4.1 核心概念：果品检验台，不是田园风画册

关键词：**清晰、鲜活、可判断、有触感、不孩子气**。

视觉来源应该是真实的检果场景：果面图像、分级色卡、批次标签、称重数字、风险标记和流向箭头。不再使用暖米色＋宋体＋印章作为主要识别。

### 4.2 建议 token

| 角色 | 建议值 | 用途 |
| --- | --- | --- |
| Canvas | `#F2F4EF` | 全局底色，冷中性 |
| Surface | `#FFFFFF` | 工作面与商品信息 |
| Ink | `#17211B` | 主文字 |
| Leaf | `#1F6B45` | 主操作、通过、可信信息 |
| Apple | `#D94A36` | 高风险、禁售、关键提醒 |
| Harvest | `#E5A62F` | 待处理、需复核 |
| Sky | `#DDEAF0` | 中性说明、AI 辅助信息 |
| Border | `#D7DED6` | 结构分隔 |

不将紫色作为主色。虽然 UI/UX Pro Max 检索推荐了紫色市集方案，但它与农产品检验和当前品牌语义不匹配。

### 4.3 字体系统

- 标题：现代中文黑体，优先评估 MiSans / HarmonyOS Sans SC，正式引入前确认授权。
- 正文：系统中文无衬线字体，保证农户手机上的读取速度。
- 等级、批次和数据：使用等宽数字字体，让“A / B / C / 禁售”具有检验编码感。
- 删除首页独立宋体，避免进入功能页后品牌气质突然断裂。

### 4.4 唯一标志性组件：分级色带

每个商品、结果和复核任务使用一条稳定的等级色带：

```text
│ A 鲜食优选   绿
│ B 表皮瑕疵   青绿
│ C 尽快食用   黄
│ 禁售 食安风险 红
```

这条色带必须同时有文字和图形，不能只靠颜色区分。它应成为产品最容易被记住的界面语言。

## 5. 组件层级重建

后续 Agent 应先建立下列组件规则，再改页面：

1. `RoleShell`：农户、消费者、运营三种壳层，导航形态不同。
2. `StepHeader`：农户任务步骤和当前状态。
3. `GradeBand`：统一的等级/风险表达。
4. `InspectionCard`：果面图像＋瑕疵标记＋决策摘要。
5. `ProductTile`：消费者列表的大图商品卡。
6. `TaskRow`：运营待办，不再使用普通卡片。
7. `DecisionPanel`：运营决策区，固定通过/补资料/驳回的语义和按钮层级。
8. `DemoTools`：独立演示抽屉，只在演示模式出现。

禁止继续增加没有语义的 `.panel`、`.summary-box`、`.card` 变体。新组件必须回答自己对应什么业务内容。

## 6. Agent 执行阶段

### 阶段 0：基线与保护

- 保存 375px、768px、1440px 的三端截图。
- 记录农户分级、消费者意向、运营复核三条可运行路径。
- 不修改领域规则、API 契约和持久化数据结构。

验收：`npm run check` 通过，基线截图可对比。

### 阶段 1：信息架构减法（P0）

- 拆分用户任务与演示工具。
- 重组农户 4 步流程。
- 重组消费者列表与详情信息顺序。
- 将运营导航从 7 个顶部标签收敛为 5 组侧导航。
- 删除用户端中的开发词汇。

验收：任意页面首屏只有一个主操作；5 秒内能说出当前状态和下一步。

### 阶段 2：设计 token 与核心组件（P0）

- 建立颜色、字号、间距、圆角、边框、阴影和动效 token。
- 实现 `GradeBand`、`StepHeader`、`InspectionCard`、`ProductTile`、`TaskRow`、`DecisionPanel`。
- 删除重复和相互覆盖的 legacy CSS，不再通过文件末尾叠加覆盖实现新风格。

验收：相同业务含义在三端使用同一套 token；页面不再全由同款白卡片组成。

### 阶段 3：农户端重设（P0）

- 先改拍照页，再改信息页、结果页和状态页。
- 移除默认样本墙和技术 JSON。
- 优化大拇指区域、拍摄指导、等待状态和风险结果。

验收：用户在不看说明文档的情况下，可从选图走到结果；每屏不超过一个主 CTA。

### 阶段 4：消费者端重设（P0）

- 重做大图商品列表和首屏信任详情。
- 建立等级色带和透明价格说明。
- 合并重复反馈入口，明确意向与真实订单的边界。

验收：详情首屏可回答价格、瑕疵、安全和建议用途；主操作唯一。

### 阶段 5：运营端重设（P1）

- 建立桌面工作台壳层。
- 待办优先，KPI 收敛，完成复核三栏工作区。
- 将坏例、Eval、运行追踪组成一个 AI 质量工作区。

验收：运营打开首页即可看到最高风险待办；完成一条复核不需在多个页面之间来回切换。

### 阶段 6：状态、动效和无障碍（P1）

- 补齐空、加载、成功、失败、禁用、超时、无网络状态。
- 只在状态转换时使用 160–240ms 动效；不对所有卡片使用浮起效果。
- 确保 44px 点击区、键盘焦点、减少动效和对比度。

验收：键盘可完成主流程；色弱用户不依靠颜色也能理解等级和风险。

### 阶段 7：回归验收（P0）

- 对比 375px、768px、1440px 截图。
- 重跑农户、消费者、运营三条主流程。
- 检查页面级横向滚动、粘性元素遮挡和浏览器返回。
- 运行 `npm run check`，检查控制台错误。

## 7. 后续 Agent 必须遵守的规则

1. 每次只执行一个阶段，阶段通过验收后再进入下一阶段。
2. 先改结构和文案，后改颜色、阴影和动效。
3. 不得通过在 CSS 末尾继续叠加大片覆盖完成重设；应删除或合并旧规则。
4. 不得改变分级规则、API 字段、持久化数据结构和已有测试语义。
5. 不使用紫色科技渐变、无意义玻璃拟态、大量圆角白卡或 emoji 图标。
6. 每个页面修改前写清：用户是谁、当前任务是什么、唯一主操作是什么。
7. 每个阶段结束必须提供：修改文件、前后截图、交互路径、测试结果和剩余风险。

## 8. 总验收标准

- 农户首屏不再出现 Mock、样本墙、生成演示数据和重置。
- 消费者首屏不再出现开发或运营操作。
- 运营首屏的第一个内容区是待办，不是指标卡墙。
- 三端都有明确的当前位置、主任务和下一步。
- A/B/C/禁售在列表、详情、结果和复核页使用统一的等级色带。
- 主要页面不再由同一种米白圆角卡片重复堆叠组成。
- 375px 无页面级横向滚动，1440px 运营端利用完整宽度。
- 三条主流程可用键盘和触屏完成，并有明确的加载、成功和失败反馈。


## 9. 参考图风格转译：鲜活果品实验室

参考图中的青柠绿、果皮/玻璃纹理、粗黑中文和版画水果插图可作为新的品牌语言，但必须从 PPT 封面风格转译为可操作的产品界面。

### 9.1 使用强度

| 场景 | 风格强度 | 用法 |
| --- | ---: | --- |
| 角色首页/品牌层 | 80% | 大字标题、青柠光斑、版画苹果、不对称构图 |
| 空状态/分级完成 | 40% | 局部纹理、分级大字、短暂状态转场 |
| 农户上传与表单 | 20% | 青柠主按钮、步骤进度、少量品牌图形 |
| 消费者商品 | 15% | 等级色带、大图、价格与瑕疵对比 |
| 运营工作台 | 5% | 只保留品牌色、粗标题和活动导航 |

### 9.2 更新后的 token

Lab Lime `#C9F24A`、Lime Light `#EAF8A8`、Forest `#174F13`、Canvas `#F4F6EE`、Surface `#FFFFFF`、Ink `#17211B`、Apple `#D94A36`、Harvest `#E3A327`、Border `#D8DED2`。

青柠色不承担全部状态含义；A/B/C/禁售仍使用文字、图形和独立风险色共同表达。

### 9.3 图形、纹理与字体规则

- 版画苹果用于品牌首页、空状态和完成状态，不替代真实果面图像。
- 纹理只出现在 hero 或局部强调区；表单、正文和运营数据背后必须使用纯色工作面。
- 青柠底使用 Forest/Ink 文字，不使用白色小字；正文对比度不低于 4.5:1。
- 品牌标题使用粗黑中文无衬线，删除首页宋体。
- 手机端 hero 标题控制在 36–44px，确保首屏仍能看到主操作。
- 强调性装饰只保留果皮光斑纹理，不叠加印章、玻璃卡和渐变光球。
