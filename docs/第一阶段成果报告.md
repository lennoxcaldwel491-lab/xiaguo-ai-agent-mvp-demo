# 瑕果智选 AI Agent MVP 第一阶段成果报告

## 1. 阶段目标

第一阶段目标不是做完整电商平台，而是验证一条最小业务闭环：

```text
农户提交苹果图片和基础信息
-> AI Agent 初步识别瑕疵并分级
-> 系统生成上架说明和消费者文案
-> 人工复核高风险商品
-> 低风险商品进入消费者页
-> 消费者模拟下单或反馈
-> 反馈和误判进入坏例池
```

当前这一闭环已经完成，并已从纯 mock 逻辑升级为可接入 Qwen-VL 真实视觉 API 的版本。

## 3. 已具备的功能

### 3.1 农户上架端

已实现：

- 选择苹果样本图片。
- 上传本地苹果图片。
- 为上传图片选择临时标签，用于 mock Agent 测试。
- 填写农户基础信息：
  - 水果品类
  - 产地
  - 重量
  - 采摘时间
  - 期望售价
  - 农户备注
- 点击“开始智能分级”触发 Agent。

当前设计原则：

- 农户填写字段尽量少，降低上架门槛。
- 图片质量检查第一版先通过提示和复核兜底，不做复杂图像质检。

### 3.2 AI Agent 分级

已实现两种模式：

#### Mock 模式

使用图片目录标签或上传时选择的临时标签，模拟视觉识别结果。

```text
fresh -> A 级，可上架
scab_defect -> B 级，可上架
bruise_defect -> C 级，建议复核
rot_defect -> 禁售，强制复核
```

#### API 模式

通过本地 `server.js` 调用 Qwen-VL / 阿里百炼视觉模型。

当前 API 能力：

- 读取 `.env.local` 中的 DashScope API Key。
- 把本地图片转为 base64 data URL。
- 调用 Qwen-VL 视觉模型。
- 要求模型输出固定 JSON。
- 对模型输出做字段归一化。
- 无 API Key 或调用失败时回退 mock，保证演示不中断。

Agent 输出字段包括：

- `fruit_type`
- `defect_type`
- `defect_label`
- `business_defect`
- `grade`
- `confidence`
- `edible_safety`
- `safety_label`
- `price_suggestion`
- `farmer_explanation`
- `consumer_copy`
- `review_required`
- `risk_flags`
- `next_action`

### 3.3 分级报告页

已实现：

- 展示建议等级。
- 展示瑕疵类型。
- 展示置信度。
- 展示是否需要复核。
- 展示农户可读解释。
- 展示完整结构化 JSON。
- 支持复制 JSON。
- 支持下载 `grading_report.json`。

### 3.4 农户确认动作

已实现：

- 低风险商品可确认上架。
- 高风险商品或低置信度商品提交人工复核。
- 农户可返回修改基础信息。

当前限制：

- 第一版不允许农户自行把 AI 判断等级大幅调高。
- 等级争议进入复核，而不是让农户直接修改。

### 3.5 消费者商品页

已实现：

- 展示已上架商品列表。
- 展示商品详情。
- 展示等级、产地、重量、价格。
- 解释“为什么便宜”。
- 说明是否影响食用。
- 展示 AI 生成的透明商品说明。
- 提供售后保障说明。
- 支持模拟下单。
- 支持消费者反馈。

消费者反馈类型包括：

- 我愿意购买
- 我还担心食用安全
- 说明不够清楚
- 价格不够有吸引力
- 不接受这种瑕疵

### 3.6 运营后台

已实现：

- 人工复核队列。
- 商品管理表。
- 复核通过。
- 驳回并记录坏例。
- 展示商品状态：
  - listed
  - pending_review
  - rejected

当前复核原则：

- 高风险优先。
- 腐烂、霉变、破皮渗液必须复核。
- 低置信度必须复核。
- 人工复核结果可进入坏例池。

### 3.7 规则与坏例池

已实现：

- 苹果分级规则展示。
- 坏例池展示。
- 消费者负向反馈可进入坏例池。

坏例字段包括：

## 4. 当前产品逻辑

### 4.1 正常上架路径

```text
农户选择 fresh 或 scab_defect 图片
-> 填写基础信息
-> Agent 判断为 A/B 级
-> review_required=false
-> 农户确认上架
-> 商品进入消费者页
-> 消费者查看说明
-> 模拟下单或反馈
```

适用场景：

- 无明显瑕疵苹果。
- 表皮果锈、疮痂斑，但不影响果肉食用的苹果。

### 4.2 人工复核路径

```text
农户选择 bruise_defect 图片
-> Agent 判断为 C 级
-> review_required=true
-> 商品进入复核队列
-> 运营查看图片和 AI 说明
-> 运营通过或驳回
-> 通过后进入消费者页
-> 驳回后进入坏例池
```

适用场景：

- 轻微碰伤。
- 置信度不足。
- 模型无法明确判断。

### 4.3 高风险拦截路径

```text
农户选择 rot_defect 图片
-> Agent 判断为 blocked 或高风险
-> review_required=true
-> next_action=manual_review
-> 商品进入复核队列
-> 原则上不能直接展示给消费者
-> 运营驳回或要求补图
-> 样本进入坏例池/回归测试
```

适用场景：

- 疑似腐烂。
- 疑似霉变。
- 破皮渗液。
- 食品安全风险。

### 4.4 反馈回流路径

```text
消费者查看商品页
-> 提交购买意向或顾虑反馈
-> 正向反馈记录购买意向
-> 负向反馈进入坏例池
-> 运营复盘说明、价格或规则
```

适用场景：

- 消费者不信任瑕疵说明。
- 消费者认为价格吸引力不足。
- 消费者不接受某类瑕疵。

## 6. AI 与人工边界

### AI 负责

- 初步识别瑕疵类型。
- 给出等级建议。
- 给出置信度。
- 判断是否需要复核。
- 生成农户解释。
- 生成消费者商品说明。
- 输出结构化 JSON。

### 人工负责

- 食品安全最终判断。
- 高风险商品复核。
- 等级争议处理。
- 规则调整。
- 坏例复盘。
- 是否允许上架的最终责任。

### 硬性安全规则

- 疑似腐烂必须复核。
- 疑似霉变必须复核。
- 破皮渗液必须复核。
- 置信度低于 0.7 必须复核。
- AI 不直接决定食品安全结论。
- 消费者文案不能承诺绝对安全。

## 7. 当前技术结构

文件：

```text
index.html

app.js
```

特点：

- 静态页面。
- 无需复杂框架。

### 本地服务

文件：

```text
server.js
```

能力：

- 静态资源服务。
- `/api/health`：检查 API 配置。
- `/api/grade`：调用 Qwen-VL 或 fallback mock。
- 读取 `.env.local`。
- 本地图片转 base64 data URL。

### 配置

文件：

```text
.env.local
.env.example
```

示例：

```text
VISION_PROVIDER=qwen
DASHSCOPE_API_KEY=你的 DashScope API Key
DASHSCOPE_MODEL=qwen-vl-max
DASHSCOPE_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
PORT=8787
```

## 8. 当前成果价值

第一阶段已经证明：

- 产品闭环成立。
- 农户上架流程可以简化。
- AI Agent 可以作为分级初判角色。
- 人工复核能兜住高风险商品。
- 消费者侧可以通过透明说明建立信任。
- 坏例池可以形成规则迭代闭环。
- 项目已经具备真实视觉模型接入能力。

这意味着项目已经从“想法”推进到“可演示 MVP”。

## 9. 当前限制

- 当前真实 AI 识别效果还未系统评估。
- 只有苹果单品类。
- 样本数量只有 12 张演示样本。
- 还没有人工标注集。
- Eval 还没有计算真实准确率。
- 没有真实订单、支付、物流。
- 没有农户账号和权限系统。
- 没有真实售后流程。
- 没有多角色登录。

这些限制是 MVP 阶段可接受的，因为当前重点是验证 AI 分级和信任说明闭环。

## 10. 下一阶段建议

下一阶段不建议继续盲目加页面，而应重点验证真实 AI 可靠性。

优先做：

1. 用真实 API 跑完 12 张样本。
2. 建立 40 张苹果人工标注集。
3. 做真实 AI Eval。
4. 统计腐烂风险漏判率。
5. 优化提示词和强制复核规则。
6. 把坏例回归测试流程做完整。

关键验收指标：

- 腐烂风险漏判率：0。
- 高风险样本复核召回率：100%。
- JSON 可解析率：100%。
- 等级一致率：先以 70% 作为 MVP 参考线。
