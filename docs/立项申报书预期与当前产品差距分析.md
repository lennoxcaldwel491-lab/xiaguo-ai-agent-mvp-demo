## 1. 总体判断

当前产品已经完成的是一个 **苹果单品类、演示级、AI Agent 辅助分级与可信上架 MVP**。它跑通了：

```text
农户上传/选择苹果图片
-> AI 初步分级
-> 生成分级报告和商品说明
-> 高风险进入人工复核
-> 可上架商品进入消费者页
-> 消费者模拟下单/反馈
-> 负反馈和误判进入坏例池
-> Eval 页面做小样本评测
```

但立项申报书中的目标更大，预期是：

```text
消费者调研 + 农户访谈
-> 多品类分级标准
-> 图像数据集搭建与模型训练
-> AI Agent 智能定级
-> 微信小程序/电商交易平台
-> C 端零售 + B 端加工 + 多渠道供需匹配
-> 物流协同 + 售后保障 + 传播推广
-> 试点运营 + 数据沉淀 + 规模化推广
```

因此，两者的差距本质是：

> 当前产品完成了“可信分级与上架闭环 Demo”，尚未完成“多品类、多端口、多渠道、可运营的智能分级电商平台”。

需要注意：按申报书进度安排，2026 年 5 月当前阶段原本主要对应“烟台产区农户访谈、初步构建分级标准框架”。小程序核心功能、模型训练、试点入驻、营销模块等多数内容在申报书中本来就是 2026 年 7 月至 2027 年 1 月之后的计划。因此，下面的“未实现”不全部等同于延期，也包括当前 MVP 主动收敛后暂未覆盖的长期预期。

## 4. 按产品模块拆解差距

### 4.1 分级标准与数据集

申报书预期：

- 构建适配不同水果的分级规范。
- 明确优果、A/B/C 级瑕疵果判定规则。
- 量化外观、尺寸、瑕疵等核心参数。
- 建立标准化水果分级数据集。
- 使用实地采样、数据清洗、筛选与标注。
- 形成《水果分级标准（正式版）》。

当前状态：

- 仅针对苹果做了 4 类演示标签：
  - `fresh`
  - `scab_defect`
  - `bruise_defect`
  - `rot_defect`
- 当前业务映射是：
  - fresh -> A 级
  - scab_defect -> B 级
  - bruise_defect -> C 级/建议复核
  - rot_defect -> 禁售/强制复核
- 已抽取 12 张苹果样本进入 MVP。
- 还没有 40 张以上人工标注集。
- 还没有正式的多品类分级标准手册。
- 还没有量化瑕疵面积、尺寸、缺陷比例等指标。

未实现功能：

| 未实现项 | 说明 | 建议优先级 |
| --- | --- | --- |
| 多水果分级标准 | 当前只有苹果，未覆盖芒果、草莓等 | P2 |
| 优果 + A/B/C 统一规则 | 当前规则偏 Demo，不是正式标准 | P0 |
| 标准手册 | 没有形成可交付的《水果分级标准》文档 | P0 |
| 人工标注集 | 缺少可计算准确率的人工标注数据 | P0 |
| 缺陷面积/比例量化 | 未基于图像分割计算缺陷占比 | P2 |
| 复杂拍摄环境样本 | 缺少多光照、多背景、多角度样本 | P1 |

判断：这是当前与申报书差距最大的基础模块。当前 Demo 是“规则映射 + 视觉 API 初判”，还不是申报书里的“标准化数据集 + 量化分级标准 + 模型训练”。

### 4.2 AI 识别算法与模型训练

申报书预期：

- 基于 YOLOv5/v8 做水果外观瑕疵识别。
- 使用 OpenCV、PyTorch、YOLO、UNet 或轻量图像分割模型。
- 做水果轮廓分割、缺陷区域识别、面积比例计算。
- 完成 AI 模型初步训练。
- 持续优化模型识别精度与推理速度。
- 针对小面积/隐蔽瑕疵做数据增强和模型微调。

当前状态：

- 已接入 Qwen-VL / 阿里百炼视觉模型。
- 后端 `server.js` 可将本地图片转 base64 data URL 并调用 API。
- 支持固定 JSON 输出和字段归一化。
- 无 API Key 或 API 失败时回退 mock。
- 没有自研 YOLOv5/v8 模型训练。
- 没有 OpenCV 图像处理链路。
- 没有图像分割、缺陷面积占比计算。
- 没有真实准确率评估。

未实现功能：

| 未实现项 | 当前替代方案 | 风险 |
| --- | --- | --- |
| YOLOv5/v8 自研模型 | Qwen-VL 通用视觉模型 | 可解释性和可控性弱于专用模型 |
| 训练数据集 | 12 张演示样本 | 样本量不足，无法证明泛化 |
| 图像分割/缺陷面积计算 | 模型文字判断 | 无法形成量化分级依据 |
| 模型准确率报告 | MVP Eval 统计 | 无人工标准答案，不能算真实准确率 |
| 推理性能优化 | 本地 Node 调 API | 未验证并发、延迟和稳定性 |
| 小面积瑕疵识别 | 通用视觉识别 | 容易漏判隐蔽风险 |

建议：短期不急于训练模型，先完成人工标注集和 Eval；中期再判断是否需要 YOLO/分割模型。否则容易先做算法，但没有产品验证闭环。

## 6. 当前产品最主要的 10 个缺口

### 缺口 1：只有苹果单品类，未覆盖多品类

申报书希望覆盖烟台苹果、海南芒果等多产区、多品类。当前只做苹果。  
短期建议：继续只做苹果，但把苹果规则和 Eval 做扎实。多品类放到 P2。

### 缺口 2：没有正式人工标注集

当前只有 12 张演示样本，无法证明 AI 精度。  
短期建议：先做 40 张苹果人工标注集，每类 10 张。

### 缺口 3：没有自研视觉模型训练

申报书写了 YOLOv5/v8、OpenCV、PyTorch、UNet 等技术路线。当前是 Qwen-VL API。  
短期建议：不要立即训练模型，先用 Eval 判断通用视觉模型是否足够支撑 Demo。

### 缺口 4：没有智能定价模型

当前只有价格建议区间，未接市场价格和供需数据。  
短期建议：做规则定价表即可，如 A=市场价 90%-100%，B=65%-80%，C=50%-65%。

### 缺口 5：没有微信小程序

申报书预期是微信小程序，当前是 Web Demo。  
短期建议：先把 Web Demo 打磨成应用形态，再迁移小程序。

### 缺口 6：没有真实交易系统

当前只有模拟下单。没有订单、支付、物流、售后。  
短期建议：Demo 阶段保留模拟下单，但补充订单数据对象和状态说明。

### 缺口 7：没有 B 端和商超端

申报书的 C/B 双链路未实现。  
短期建议：先在文档和规则中加入“C 级加工果流向 B 端”的设计，不急着做完整页面。

### 缺口 8：没有物流协同

申报书提到第三方物流、产地集配。当前未实现。  
短期建议：Demo 中只展示“物流暂不接入”的边界，避免扩大范围。

### 缺口 9：没有传播推广功能

短视频、公众号、一键分享、助农案例都未实现。  
短期建议：先做“助农案例/减损说明”静态模块，为后续传播准备素材。

### 缺口 10：没有真实试点和运营数据

当前没有农户、消费者、加工企业真实使用数据。  
短期建议：先设计试点数据采集表，明确未来要收哪些指标。

## 7. 哪些不建议现在做

从 MVP 角度，以下功能虽然在申报书中出现，但当前不建议马上实现：

| 功能 | 不建议当前做的原因 |
| --- | --- |
| 真实支付 | 合规、退款、履约复杂度高，不能直接验证分级信任 |
| 完整物流 | 需要外部资源，不适合本地 Demo 阶段 |
| 多品类水果 | 会扩大规则和数据集成本，影响苹果验证 |
| 自研 YOLO 模型训练 | 没有足够标注数据前，训练结果不可验证 |
| B 端采购完整系统 | 需要真实商家需求和供应链数据 |
| 优惠券/拼团 | 偏运营增长，不是当前核心验证 |
| 规模化推广 | 需要先有稳定产品和真实试点 |

当前最应该优先做的是：

```text
苹果单品类规则标准化
-> 40 张人工标注集
-> 真实 AI Eval
-> 高风险护栏
-> 前后端应用化
-> 复核/坏例/指标闭环
```
